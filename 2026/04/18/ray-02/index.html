Ray Actor Pool 实战：GPU 模型预热与节点级资源调度

发表于 2026-04-18 分类于 Python

本文仅人工调整个别地方，肉眼可见一股渣滓味。

一、概述

在上一篇Ray 应用场景一例文章中，介绍了使用 Ray 内置的 node:IP 资源来实现任务绑定到特定节点。这种方法简单直接，不过 node:IP 在 Ray 官方文档中也没体现，可能过于”私有”。另外在设计层面上还有个缺陷：每次 GPU 任务启动时，都需要重新加载模型，在某些应用场景下纯属浪费。

本文介绍一种更高效的方案：使用 Actor Pool 实现模型预热，并通过 NodeAffinitySchedulingStrategy 实现节点级资源调度。

方案 模型预热 调度灵活性 实现复杂度
node:IP 资源绑定 ❌ 每次重新加载 ⭐⭐ 简单
Actor Pool + NodeAffinity ✅ 预加载复用 ⭐⭐⭐⭐ 中等

核心思路：

  1. Actor 预热：在 Actor 初始化时加载模型，Actor 持久存活，跨任务复用
  2. 节点级 Actor Pool：每个节点独立管理自己的 Actor Pool，避免跨节点数据传输
  3. 零资源 Manager：用一个不占 CPU/GPU 的 Actor 管理其他 Actor

二、设计要点

1、为什么需要 Actor Pool？

Ray 提供了 ray.util.actor_pool，但官方文档明确指出它不是线程安全的。在多任务并发场景下，直接使用会导致竞态条件。

更关键的是，我们需要 Actor 具备以下特性：

特性 说明
模型预热 Actor 初始化时加载 Sherpa（ASR）和 MiniCPM（视频分析）模型，约 30 秒
持久存活 lifetime="detached"，Actor 不随任务结束而销毁
节点绑定 通过 NodeAffinitySchedulingStrategy 确保 Actor 不会跨节点迁移
排队机制 满载时新请求自动排队，无需调用方处理

2、为什么用 Manager Actor 管理其他 Actor？

如果直接创建多个 GPU Actor，调用方需要：

  1. 知道有多少个 Actor
  2. 知道每个 Actor 的名字
  3. 自己实现调度逻辑（哪个 Actor 空闲？）
  4. 自己管理空闲/忙碌状态

三、Talk is cheap

1、RayActorPoolManagerActor

零资源 Manager Actor，管理 Actor Pool：

关键点：

  • num_cpus=0, num_gpus=0：不占用计算资源，纯调度角色
  • asyncio.Queue：线程安全的队列，满载时自动阻塞
  • NodeAffinitySchedulingStrategy(soft=False)：硬绑定到指定节点
  • lifetime="detached"：Actor 持久存活，跨任务复用

2、RayActor

GPU Worker Actor，预加载模型：

关键点：

  • num_gpus=1：每个 Actor 独占一个 GPU
  • 模型预热：__init__ 中加载模型，避免每次任务重新加载
  • Actor 数量 = GPU 数量，不超卖资源

四、对比 node:IP 资源绑定方案

特性 node:IP 资源 NodeAffinity + Actor Pool
数据本地性 ✅ ✅
模型预热 ❌ 每次重新加载 ~30s ✅ 初始化时加载一次
Actor 复用 ❌ ✅ 持久存活
排队机制 需自己实现 ✅ asyncio.Queue 自动处理
调度灵活性 ⭐⭐ ⭐⭐⭐⭐

上一篇文章的方案适合简单场景，Actor Pool 方案适合高频、长期运行的生产环境。
